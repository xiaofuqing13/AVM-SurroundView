1. 引言
1.1 编写目的
本测试报告依据国家标准 GB/T 9386-2008《计算机软件测试文档编制规范》，详细记录对 "AVM 360° 全景环视系统"的系统性测试过程。旨在通过规范的测试流程验证软件功能完整性、评估质量特性、记录缺陷修复情况，为项目验收、后续迭代及质量追溯提供标准化技术文档支撑。具体目标包括：
功能验证：确认软件实现需求规格说明书定义的全部功能项，确保无需求遗漏或逻辑错误。
质量评估：从功能正确性、界面可用性、系统稳定性、性能效率等维度建立量化评估体系。
缺陷管理：遵循缺陷生命周期管理流程，记录缺陷发现、定位、修复、验证的完整轨迹。
决策支持：为项目评审组提供客观的质量数据，作为软件交付、版本发布的核心依据。
1.2 项目背景
"AVM 360° 全景环视系统" 是面向智能网联汽车的核心辅助驾驶系统，通过车身四周部署的多路摄像头采集环境数据，经图像拼接、畸变校正等处理后，在 GUI 界面呈现实时鸟瞰视图及多角度监控画面。本软件作为人机交互核心模块，承担视频流实时渲染、视图切换控制、辅助线动态绘制等关键功能，其交互效率与显示效果直接影响驾驶安全性。本次测试对象为竞赛交付版本，重点验证 UI 布局合规性、功能逻辑正确性及多工况下的运行稳定性。

2.3 测试方法
为确保测试的全面性与深度，本项目综合运用多种测试方法，构建覆盖 “静态到动态”“前端到后端” 的立体化测试策略。
1. 静态测试（Static Testing）
在不实际运行代码的情况下，通过评审和审查发现早期缺陷：

•	需求规格评审：逐条审查《需求规格说明书》，检验需求的明确性、一致性与可测试性，为测试用例设计奠定基础。
•	UI/UX 设计审查：将 GUI 实际布局与设计图比对，检查像素级偏差、元素对齐、字体规范及图标使用是否符合设计标准。
•	代码规范审查（部分）：由开发自测阶段完成，检查代码是否规范。
2. 动态测试（Dynamic Testing）
通过实际运行软件，验证其在不同条件下的行为表现：
2.1 黑盒测试（Black-box Testing）
•	核心方法：作为功能测试的主要手段，测试人员从用户视角出发，不关注内部实现，验证功能是否符合需求。
•	用例设计技术：
•	等价类划分：将输入数据划分为有限的有效 / 无效等价类，覆盖典型场景。例如，分辨率输入测试有效值（1280×720）。
•	边界值分析：针对有范围的输入（如畸变校正滑块），重点测试边界值（最小值、最大值、略超边界值），发现边界处理错误。
•	用例规模：基于上述方法设计 25 个功能及非功能测试用例，详细清单见附录。
2.2 灰盒测试（Gray-box Testing）
•	核心方法：测试人员了解程序内部结构，结合界面操作与后台逻辑状态进行测试。
•	应用场景：
•	线程同步验证：在 main_gui.py 中高频切换视图、勾选效果，同时通过后台日志或调试器监控四个摄像头采集线程、处理线程和鸟瞰图拼接线程的数据同步，排查死锁或竞争条件。
•	资源利用率监控：执行特定功能（如开启图像增强）时，通过任务管理器等工具监控 CPU 和内存变化，验证资源消耗是否符合预期。

6. 缺陷统计与分析
6.1 典型缺陷分析 
BUG-01
•	问题描述：程序启动后，主界面各视频窗口均显示空白或黑色，不刷新实时画面。控制台报错 AttributeError: 'MainWindow' object has no attribute 'update_frames'
•	根本原因：代码合并时误删视频帧更新函数，导致 UI 线程无法获取图像数据
•	修复方案：恢复被删除的 update_frames () 方法，为确保多线程数据安全，在 update_frames() 方法内部访问共享的图像缓冲区（如self.birdview.get()）前后，增加适当的线程同步锁
BUG-02
•	问题描述：在主界面勾选“色彩校正”复选框后，鸟瞰图画面无任何视觉变化，该功能未生效
•	根本原因：在 surround_view/birdview.py 的 run() 方法主循环中，函数调用顺序存在逻辑错误。代码先执行了 make_white_balance() 对 上一帧的旧图像 进行色彩校正，但紧接着调用 stitch_all_parts() 使用 当前的新图像 完全覆盖了处理结果，导致校正效果被丢弃
•	修复方案：调整 run() 方法内的代码顺序，将 make_white_balance() 的调用移动到 stitch_all_parts() 之后、copy_car_image() 之前。确保色彩校正操作作用于最新拼接完成的鸟瞰图上
BUG-03
•	问题描述：当侧方摄像头存在桶形畸变（需要负值系数校正）时，GUI提供的畸变校正滑块最小值被限制为0，导致用户无法向负值方向调整，无法有效校正该类畸变
•	根本原因：左/右摄像头图像边缘的桶形畸变无法被消除，导致画面边缘物体变形，影响侧方视野的准确性
•	修复方案：将GUI滑块的取值范围从 [0, 45] 调整为 [-45, 45]，允许用户进行双向调整
BUG-04
•	问题描述：当物理连接的摄像头数量少于4个，或任一摄像头的设备ID与 main_gui.py 中硬编码的 camera_ids = [0, 3, 2, 4] 列表不符时，程序启动时直接崩溃
•	根本原因：代码在进入核心初始化流程（创建 CaptureThread）之前，没有对硬件环境的完备性进行前置检查。它乐观地假设所有在 camera_ids 列表中定义的摄像头都已正确连接并可用。因此，当后续代码尝试为一个不存在的设备ID创建视频捕捉对象(cv2.VideoCapture)并启动线程时，会因对象初始化失败而引发致命错误，导致程序崩溃
•	修复方案：增加启动前置检查: 在 setup_surround_view 方法的入口处，增加了一个摄像头连接预检查循环。该循环会遍历 camera_ids 列表，尝试为每个ID创建一个临时的 cv2.VideoCapture 对象，以验证其是否可以被成功打开。如果预检查发现连接的摄像头数量少于列表要求的数量，程序会利用 QMessageBox.critical 弹出一个模态错误对话框。该对话框会清晰地告知用户“需要几个摄像头”、“实际检测到几个”以及“具体哪个ID的摄像头缺失”。在用户关闭对话框后，setup_surround_view 方法返回 False 信号，主程序的 __init__ 方法接收到此信号后会中止后续的初始化（如启动QTimer），从而使程序停留在安全状态，避免了崩溃。
